# Confidence Quest — Техническое задание (PRD) v3.2

> **Платформа:** iOS / Android
> **Стек:** Flutter + SQLite (Local) + Supabase (Backend/Auth/Storage)
> **Тип монетизации:** Freemium (PRO-подписка)
> **Версия:** v3.2 — безопасная defer-модель вместо punitive fail

**Главное отличие от v2:** продукт больше не является UGC/video/social-feed продуктом. Убраны публичная Arena, пользовательские видео, лайки, комментарии, Hall of Fame, видео-босс-рейды, приоритетная модерация и любые PRO-фичи, завязанные на охват. Продукт теперь — приватная mobile RPG для безопасной прокачки социальной уверенности через реальные мини-квесты, подготовку и короткую рефлексию.

**Что изменено в v3.2 по сравнению с v3.1:**

1. Пользовательское действие «Сбежать» заменено на нейтральное «Отложить».
2. Отложенный квест не списывает энергию, не создаёт глобальный cooldown и может быть сразу начат снова.
3. NPC-подготовка и выпавший модификатор сохраняются после откладывания.
4. Добавлено отдельное событие `quest_deferred`; `quest_failed` оставлено только для timeout/system interruption и совместимости со старыми событиями.
5. Из схем нового MVP удалено `cooldown_until`; миграция старого состояния снимает уже созданную блокировку.

**Что изменено в v3.1 по сравнению с v3.0:**

1. Удалена премиум-валюта `crystals` из всех схем — без публичной экономики она не нужна.
2. Смягчён первый онбординг-квест: вместо комплимента незнакомцу — тёплое «спасибо, хорошего дня».
3. Event log разделён на MVP-ядро (append-only + дедупликация + базовая валидация) и Post-MVP (full rebuild, anomaly detection, soft-shadowban).
4. PRO-кубик ограничен: 3 броска в день или 1 reroll на каждый квест. Безлимита нет — иначе кубик перестаёт быть событием.
5. Boss Raid получил обязательный low-pressure вариант с тем же XP — чтобы не давить FOMO на тревожных пользователей.
6. Добавлен раздел «Ethical Boundaries / Not medical advice».
7. Roadmap изменён с жёстких 8 недель на диапазон: 8–10 недель для команды (Flutter + backend + контент), 10–12 недель для solo/small team.

**Цель v3.2:** сделать MVP, который реально можно разработать за 8–12 недель в зависимости от размера команды, выпустить в App Store / Google Play без юридических и модерационных рисков, и проверить гипотезу core loop на живых пользователях.

## 1. Позиционирование и Суть

**Проблема.** Трекеры привычек скучные. Приложения для ментального здоровья стигматизируют — их стыдно показать друзьям. Между знанием «мне нужно прокачать социализацию» и действием нет моста. Молодые люди 18–30 избегают социальных ситуаций из-за тревоги, но не готовы идти к терапевту и не хотят использовать продукты с ярлыком «у меня проблема».

**Решение.** Приватная mobile RPG, где реальные социальные действия — это квесты. Пользователь получает 3 daily-квеста разной сложности, выбирает один, при необходимости проходит NPC-тренировку, выполняет действие в реальности, подтверждает выполнение, отвечает на 3 коротких reflection-вопроса и получает XP. Никакого публичного контента, никаких лайков, никакой социальной нагрузки. Прогресс принадлежит только пользователю.

**Позиционирование.** Confidence Quest — social-confidence RPG. Это не dating app, не PUA-тренажёр, не therapy app. Маркетинг бьёт в образ «прокачиваю социальную уверенность как в игре», а не «учусь клеить» и не «лечу тревожность». Тон — спокойный, уважительный, поддерживающий.

**Бренд.** Confidence Quest как основное название. Слово «Rizz» остаётся только как **внутренний стат** (один из трёх), не как идентичность продукта. В UI — «Charisma (Rizz)» в подсказке, но во всём остальном продукте используется «Charisma».

**Северная звезда:** количество выполненных оффлайн-квестов на активного пользователя в неделю. Цель MVP — ≥3. Эта метрика выбрана потому, что именно выполнение реальных социальных действий (а не Shadow-квестов и не NPC-тренировок) — единственный честный сигнал, что продукт работает.

## 2. Core Loop

Цикл одного квеста:

1. **Выдача.** В 12:00 локального времени пользователь получает 3 квеста: 2 локальных разной сложности + 1 глобальный Boss Raid.
2. **Выбор.** Пользователь выбирает один. Карточка квеста показывает текст, стоимость энергии, стат, который прокачивается, и кнопку «Бросить кубик модификаторов» (1 раз в день бесплатно).
3. **Подготовка (опционально).** Перед хардовым квестом доступна NPC-тренировка — текстовый диалог-симулятор. Прохождение даёт бафф +10% к XP.
4. **Выполнение.** Пользователь выполняет действие в реальности.
5. **Подтверждение.** Пользователь нажимает «Выполнено» или «Отложить». Никаких proof-фото или видео. Self-report.
6. **Reflection или пауза.** После «Выполнено» показываются 3 быстрых вопроса (см. раздел 7). После «Отложить» активная попытка закрывается без списания энергии и cooldown; подготовка сохраняется, к квесту можно вернуться сразу.
7. **Награда.** XP начисляется в локальную БД и логируется в event log. Энергия списывается. Прогресс-бары обновляются с анимацией.

**Длительность цикла:** 30 секунд на подготовку + реальное действие + 10 секунд на reflection. Без UGC-нагрузки.

## 3. Safety by Design

Этот раздел — главный фильтр для всего контента. Любой квест, модификатор или UI-текст должен проходить проверку по этим правилам. Квест, нарушающий хотя бы одно правило — удаляется из пула.

### 3.1. Правила дизайна квестов

1. **Не провоцировать.** Квест не должен требовать от пользователя спровоцировать незнакомца на реакцию, спор, замешательство или конфликт.
2. **Не записывать.** Квест не должен требовать фото, видео или аудиозаписи незнакомцев. Даже с согласия — это выходит за рамки MVP.
3. **Не нарушать личные границы.** Квест не должен требовать физического приближения, прикосновений, вторжения в личное пространство, подсаживания к незнакомцам.
4. **Не имитировать идентичности.** Запрещены квесты на имитацию акцента, пародию национальности, age-play, play-disability. Любая имитация чужой идентичности — запрещена.
5. **Не унижать, не пугать, не ставить в неловкость.** Квест не должен делать незнакомца объектом шутки, обязательной реакции, публичного смущения.
6. **Не давить на персонал.** Квесты на возврат товара с надуманной причиной, требование скидки без повода, спор с кассиром — запрещены. Персонал в рабочее время не может отказаться от взаимодействия, это неравные отношения.
7. **Социально допустимо.** Любое действие должно быть социально приемлемым в обычной городской среде.
8. **Уважительно.** Квест уважает время, границы и достоинство всех участников.
9. **Обратимо.** Любое действие должно быть обратимым — пользователь может в любой момент извиниться и уйти без последствий.
10. **Доступно при тревоге.** Квест должен быть выполним человеком с умеренной социальной тревожностью без профессиональной поддержки.

### 3.2. Принцип фильтрации

При создании нового квеста автор задает себе 3 вопроса:

- Хотел бы я, чтобы этот квест выполнили с моим близким человеком как «незнакомцем»?
- Если бы я был кассиром/баристой/официантом — был бы я согласен, чтобы со мной так сделали 50 раз за смену?
- Если бы журналист увидел этот квест в статье — было бы мне не стыдно?

Если хотя бы на один ответ «нет» — квест не идёт в пул.

### 3.3. Категории запрещённых квестов

| Категория | Пример | Почему запрещено |
| --- | --- | --- |
| Имитация идентичности | «Имитируй акцент» | Насмешка над национальностью/этничностью |
| Запись третьих лиц | «Сними незнакомца на видео» | Privacy / 152-ФЗ / GDPR |
| Давление на персонал | «Попроси скидку без повода» | Неравные отношения, персонал не может отказаться |
| Подсаживание | «Сядь к незнакомцам за столик» | Нарушение личных границ |
| Провокация | «Спорь с кассиром о цене» | Создаёт конфликт, страдает третья сторона |
| Публичное смущение | «Сделай что-то громкое в тишине» | Заставляет других быть невольными зрителями |
| Розыгрыш | «Разыграй незнакомца» | Не consent-ный participant |
| PUA-вайб | «Возьми номер у 5 незнакомцев» | Привлекает не ту аудиторию |

## 4. Quest System

### 4.1. Характеристики (Stats)

Три параметра, растут от разных типов квестов:

| Стат | Как прокачивается | Пример действия |
| --- | --- | --- |
| **Charisma (Rizz)** | Small talk, комплименты, тёплые фразы | Сделать искренний комплимент незнакомцу |
| **Boldness** | Инициатива, выход из зоны комфорта | Заговорить первым в очереди |
| **Networking** | Сбор информации, обмен контактами | Узнать рекомендацию у незнакомца |

### 4.2. Энергия

Максимум: 100%. Регенерация +1% каждые 10 минут.

| Триггер | Изменение |
| --- | --- |
| Лёгкий / Средний / Босс-квест | -10% / -25% / -50% |
| Естественное восстановление | +1% каждые 10 минут |
| Выполнение Shadow квеста | +5% |
| Отложил квест | 0; без cooldown |

**Safe Defer.** «Отложить» означает только «не сейчас», а не провал. Энергия не списывается, новые квесты не блокируются, NPC-подготовка и модификатор не теряются. Приложение логирует `quest_deferred` с длительностью попытки, чтобы измерять сложность квестов без наказания пользователя и без смешивания добровольной паузы с техническим failure.

**Статус «Затворник».** При энергии 0 — бейдж в UI, обычные квесты блокированы до восстановления 20% (≈3 часа). Доступны только Shadow-квесты.

### 4.3. Shadow Quests

Пассивный фарм для юзеров с 0 энергии. Выполняются онлайн, не требуют выхода из дома. Награда: +5% энергии + 50% XP от обычного квеста того же уровня. Цель — удержать привычку зайти в приложение даже в дни выгорания.

### 4.4. Кубик модификаторов

1 раз в день бесплатно. PRO — 3 броска в день + 1 reroll на каждый квест (см. раздел 8.1). Безлимитного кубика нет: модификатор должен оставаться событием, а не механикой оптимизации. Модификаторы добавляют ограничение, которое снимает «я странный потому что тревожный» и переводит в «я странный потому что так выпало»:

- **Шепот:** Сказать фразу тихо.
- **Драма:** Преувеличить эмоции на 200%.
- **Капибара:** Внедрить слово «капибара» в диалог.
- **Робот:** Говорить монотонно.
- **Эхо:** Повторить последнюю фразу собеседника.

Модификатор «Чужак / имитация акцента» — удалён. Не возвращать.

### 4.5. Пул квестов (чищеный)

50 квестов в пуле. По сравнению с v2 удалено 12 проблемных, добавлено 12 безопасных.

```sql
CREATE TABLE quests_pool (
```

## 5. Boss Raid (Global Daily Challenge)

Босс-Рейд переделан в **daily global challenge без видео**.

### 5.1. Механика

- Один общий квест дня для всех пользователей. Обновляется в 00:00 UTC.
- Пример: «Сегодня все игроки говорят баристе тёплое \"спасибо, хорошего дня\"».
- Self-report completion: пользователь нажимает «Я сделал это» — никакого proof.
- Анонимный счётчик участников: «Сегодня 12 843 игрока приняли участие». Без имён, без аватаров, без ранжирования.
- Награда: ×2 XP к обычной стоимости квеста того же уровня.
- Нет leaderboard, нет рейтинга, нет Hall of Fame, нет «топ-3».

### 5.2. Обязательный Low-Pressure вариант

У Boss Raid всегда есть два варианта выполнения с **одинаковой наградой XP**:

- **Normal:** полная версия квеста дня.
- **Low-pressure:** облегчённая версия (как в Quest Guide). Пример: если normal — «сделай искренний комплимент баристе», то low-pressure — «скажи баристе тёплое спасибо, чуть дольше обычного».

Пользователь выбирает вариант **перед** отметкой «Я сделал это». Сервер логирует выбор (`variant: normal | low_pressure`) в `quest_events` для аналитики, но XP не отличается.

**Зачем это нужно:**

- Снимает FOMO-давление: тревожный пользователь не выбирает между «выполнить страшное» и «пропустить ×2 XP».
- Не наказывает за честность: пользователь, которому сегодня тяжело, может взять low-pressure и всё равно быть «частью чего-то большого».
- Аналитика: если >60% выбирают low-pressure — это сигнал, что normal-вариант квеста слишком жёсткий, надо смягчить.

### 5.3. Зачем сохранять

Босс-Рейд без видео — это ощущение «я часть чего-то большого», без privacy/legal рисков. Это retention-хук, не социальная сеть. Пользователь видит, что он не один такой, что другие тоже сегодня делают этот квест. Этого достаточно для мотивации.

### 5.4. Схема в БД

```sql
-- Серверная таблица
CREATE TABLE daily_boss_quests (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  date DATE UNIQUE,
  level INTEGER DEFAULT 3,
  stat_type TEXT,
  energy_cost INTEGER DEFAULT 50,
  quest_text TEXT,
  low_pressure_variant TEXT NOT NULL,
  advanced_variant TEXT,
  participants_count INTEGER DEFAULT 0
);

-- При self-report completion:
-- 1. Увеличиваем participants_count через RPC
-- 2. Логируем событие в quest_events с metadata.variant = normal | low_pressure
-- 3. Начисляем XP локально
```

### 5.5. Защита от накрутки счётчика

- Один пользователь = один инкремент в день (UNIQUE constraint на `(boss_quest_id, user_id)`).
- Anti-abuse: если пользователь делает self-report без pre-logging `quest_started` события — не засчитывается.
- Допускаем небольшую долю накруток (5-10%) — это не лидерборд, точность не критична.

## 6. Quest Guide вместо Arena

Вместо публичной Arena с UGC-видео — **Quest Guide**: текстовые подсказки по выполнению каждого квеста в пуле. Кураторский контент от команды, обновляется с релизами.

### 6.1. Структура Guide

Для каждого квеста в пуле (50 штук) — отдельный гайд с 5 блоками:

1. **3 шага выполнения.** Конкретная последовательность: «1. Подойди к баристе, когда нет очереди. 2. Сделай заказ. 3. Скажи: "Кстати, спасибо за ваш кофе, всегда отличный"».
2. **Примеры фраз.** 3–5 готовых формулировок разной степени формальности.
3. **Что делать при отказе.** Сценарий, если собеседник не отреагировал или отреагировал холодно: «Это нормально. Улыбнись, забери заказ, уходи. Ты уже сделал главное — начал».
4. **Low-pressure вариант.** Облегчённая версия для тревожных пользователей: вместо «скажи комплимент незнакомцу» — «поддержи small talk с баристой, которого видишь каждый день».
5. **Advanced вариант.** Усложнённая версия для уверенных: «Сделай комплимент, который требует заметить деталь (книга, значок, футболка с принтом)».

### 6.2. UI

- Кнопка «Как выполнить?» на карточке квеста → открывает Quest Guide.
- Quest Guide доступен ДО выполнения квеста (в отличие от Reflection, который после).
- После первого выполнения квеста этого типа — Guide сворачивается в иконку.

### 6.3. Схема в БД

```sql
CREATE TABLE quest_guides (
  id TEXT PRIMARY KEY,
  quest_id TEXT,                -- FK на quests_pool
  steps_json TEXT,              -- 3 шага как JSON-массив
  phrases_json TEXT,            -- 3-5 примеров фраз
  rejection_script TEXT,        -- что делать при отказе
  low_pressure_variant TEXT,
  advanced_variant TEXT
);
```

### 6.4. Стоимость создания

50 guides × ~15 минут на гайд = ~12 часов работы продакт-райтера. Реально за 1 неделю до релиза. На порядки дешевле, чем 30-50 кураторских видео.

## 7. Reflection

После каждого выполненного квеста — 3 быстрых вопроса. Без свободного текста по умолчанию.

### 7.1. Три вопроса

1. **Насколько страшно было?** Слайдер 1–10.
2. **Как себя чувствуешь после?** Слайдер 1–10.
3. **Хочешь повторить похожий квест?** Да / Нет.

Дополнительно (опционально): текстовое поле «Заметка для себя» — пользователь может ничего не писать.

### 7.2. UX

- 3 вопроса на одном экране, без переходов.
- Заполнение занимает 5–8 секунд.
- После ответа — кнопка «Готово», начисление XP с анимацией.
- Если пользователь пропускает Reflection (кнопка «Пропустить» сверху справа) — XP всё равно начисляется. Reflection не наказываем.

### 7.3. Зачем это нужно

- **Терапевтический эффект.** Reflection закрепляет опыт и помогает пользователю заметить, что страшно было 8, а после — 9. Это контринтуитивный паттерн, который нормализует страх.
- **Аналитика.** Корреляция «страх до / состояние после» — главный сигнал о качестве квеста. Если средний «страх» по квесту 9, а «после» 4 — квест слишком жёсткий, понизить уровень или удалить.
- **Персонализация.** Если пользователь на 3-м квесте такого типа ответил «не хочу повторять» — следующий раз такой квест выдаётся с пониженной вероятностью.

### 7.4. Схема в БД

```sql
CREATE TABLE quest_reflections (
  id TEXT PRIMARY KEY,
  completed_quest_id TEXT,       -- FK на completed_quests
  fear_score INTEGER,            -- 1-10
  feel_after_score INTEGER,      -- 1-10
  want_repeat BOOLEAN,           -- true / false
  note TEXT,                     -- опционально
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 8. PRO-Подписка

PRO продаёт **личный прогресс**, а не охват. Никаких фич, завязанных на публичное внимание, ранжирование или социальный статус.

### 8.1. Что входит в PRO

| Фича | Обоснование ценности |
| --- | --- |
| 3 броска кубика в день (вместо 1) | Достаточно для экспериментов, но не превращает кубик в спам-механику |
| 1 reroll на каждый квест | Можно перебросить модификатор, если выпавший не подходит (только до начала выполнения) |
| Расширенные NPC-тренировки | Дополнительные сценарии диалогов + ветки для сложных ситуаций |
| Дополнительные quest packs | Тематические наборы квестов: «Коворкинг», «Кафе», «Мероприятия», «Семья» |
| Адаптивная сложность | Алгоритм подстраивает выдачу квестов под историю выполнений и откладываний |
| Weekly review | Короткий отчёт по неделе: что получилось, что нет, тренды страха/состояния |
| Подготовка к конкретным ситуациям | Пользователь вводит «завтра свадьба друга, надо говорить тост» — получает набор NPC-тренировок и квестов |
| Расширенная статистика | Графики прогресса по статам, корреляция страх/состояние, тренды по неделям |
| Кастомизация персонажа | Скины интерфейса, цветовые темы, иконки достижений |

#### Почему кубик не безлимитный

Безлимитный кубик в PRO превращает модификатор из «события» в «механику оптимизации». Пользователь будет рерollить, пока не выпадет самый удобный модификатор, и эффект снятия неловкости («я странный не из-за тревоги, а из-за кубика») исчезает.

Лимиты:

- **Free:** 1 бросок кубика в день.
- **PRO:** 3 броска в день (можно потратить все 3 на один квест, выбирая лучший) + 1 reroll на каждый отдельный квест.
- **В NPC-тренировке (подготовка):** кубик недоступен — модификатор применяется только к реальному квесту, не к симуляции.

Так кубик остаётся осмысленным выбором, а не фичей для reroll-спама.

### 8.2. Что явно НЕ входит в PRO

- Нет дополнительных слотов для видео (видео нет вообще).
- Нет приоритетной модерации (модерации нет вообще).
- Нет boost-ов охвата (охвата нет вообще).
- Нет ранжирования в Boss Raid (ранжирования нет вообще).
- Нет эксклюзивных наград за популярность (популярности нет вообще).

### 8.3. Ценообразование

- 299 ₽/мес или 1990 ₽/год (экономия 44%).
- Бесплатный триал — 7 дней после первого выполненного квеста (не после установки — это повышает конверт).
- StoreKit 2 на iOS, Google Play Billing Library 9.0.0 на Android (релиз 19 мая 2026).
- Server-side receipt validation через Supabase Edge Functions.

### 8.4. Принцип ценности PRO

Главная ценность PRO — помочь пользователю прогрессировать быстрее и осознаннее. Если free-версия — это 3 квеста в день и базовая NPC-тренировка, то PRO — это персональный коуч, который подбирает квесты под тебя, готовит к конкретным ситуациям и показывает твой прогресс. Это **этичная** монетизация: мы продаём инструменты развития, а не внимание или статус.

## 9. Техническая Архитектура

### 9.1. Принципиальное изменение: Event Log вместо LWW

В v2 использовался Last-Write-Wins (LWW) — клиент пушит итоговые статы, сервер перезаписывает. Это ломается при накрутках, при багах в локальной логике, при использовании на нескольких устройствах одновременно.

В v3 — **event log**: клиент логирует атомарные события, сервер хранит их иммутабельно и пересчитывает итоговые значения.

### 9.2. Схема event log

```sql
-- Локальная таблица (SQLite)
CREATE TABLE quest_events (
  id TEXT PRIMARY KEY,            -- UUID, генерируется клиентом
  event_type TEXT,                -- quest_started | quest_completed | quest_deferred | quest_failed | energy_spent | reward_granted | dice_rolled | reflection_submitted
  quest_id TEXT,
  modifier_id TEXT,
  xp_delta INTEGER,               -- изменение XP по статам
  energy_delta INTEGER,
  metadata_json TEXT,             -- дополнительные данные (fear_score, feel_after_score, etc.)
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  synced BOOLEAN DEFAULT 0        -- флаг синхронизации с сервером
);

-- Серверная таблица (Supabase, PostgreSQL)
CREATE TABLE quest_events_cloud (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id),
  event_type TEXT NOT NULL,
  quest_id TEXT,
  modifier_id TEXT,
  xp_delta INTEGER,
  energy_delta INTEGER,
  metadata JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE (user_id, id)            -- защита от дубликатов при sync
);

-- Кэшированные итоговые значения (пересчитываются сервером)
CREATE TABLE user_stats_cloud (
  user_id UUID PRIMARY KEY REFERENCES auth.users(id),
  xp_charisma INTEGER DEFAULT 0,
  xp_boldness INTEGER DEFAULT 0,
  xp_network INTEGER DEFAULT 0,
  energy INTEGER DEFAULT 100,
  last_event_id UUID,             -- указатель на последнее обработанное событие
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 9.3. Server-side recalculation

Итоговые статы (`xp_charisma`, `xp_boldness`, `xp_network`, `energy`) пересчитываются сервером одним из двух способов:

1. **Incremental.** При поступлении нового события — кэш обновляется на `xp_delta` / `energy_delta`. Подходит для 95% случаев.
2. **Full rebuild.** При рассинхроне (пользователь сменил устройство, восстановление из бэкапа) — сервер пересчитывает все итоги из полного event log за O(n).

Локально на клиенте — кэш итоговых значений для мгновенного UI. При каждом запуске приложения — fast-forward sync: клиент шлёт последние несинхронизированные события, сервер возвращает актуальный кэш.

### 9.4. Защита от накрутки — MVP vs Post-MVP

Архитектура event log правильная, но реализация делится на два этапа. Post-MVP-фичи — это перфекционизм, который может замедлить релиз MVP. Их можно добавить после валидации core loop.

#### MVP (обязательно к релизу)

- **Append-only events.** События в `quest_events_cloud` — иммутабельны. Изменить или удалить нельзя, только добавить новое (например, `quest_deferred` после `quest_started`).
- **Дедупликация.** UNIQUE constraint на `(user_id, id)` — при повторной отправке того же события (например, при ретрае sync) сервер игнорирует дубль.
- **Базовая валидация.** Сервер проверяет:
  - `quest_completed` имеет предшествующий `quest_started` (по `quest_id` в течение последних 24 часов).
  - `quest_deferred` имеет предшествующий `quest_started`, а его `energy_delta = 0`.
  - `xp_delta` соответствует квесту из `quests_pool` (а не придуман клиентом).
  - `energy_delta` не приводит к отрицательному значению.
- **Incremental recalculation.** Кэш `user_stats_cloud` обновляется на `xp_delta` / `energy_delta` при поступлении нового события. Full rebuild — только при регистрации нового устройства.

#### Post-MVP (после валидации MVP)

- **Rate limiting.** Не более N событий в минуту от одного пользователя.
- **Anomaly detection.** Фоновый job, который раз в час смотрит на подозрительные паттерны: >100 квестов в час, идеально успешные серии без откладываний, синхронные события от разных user_id.
- **Soft-shadowban.** При обнаружении аномалий — события логируются, но в `user_stats_cloud` не применяются, до ручной проверки.
- **Full rebuild.** Триггерится при рассинхроне кэша и `user_stats_cloud` — сервер пересчитывает все итоги из полного event log за O(n). Запускается либо вручную (по флагу), либо автоматически при расхождении >5%.
- **Server-side energy enforcement.** Сейчас клиент контролирует энергию локально. Post-MVP — сервер должен стать единственным источником правды для энергии, клиент только отображает.

> **Принцип:** MVP должен доказать, что event log работает на базовом уровне (дедупликация + валидация + incremental). Сложная anti-abuse-логика имеет смысл только тогда, когда есть реальная угроза накрутки — то есть когда появились публичные награды, лидерборды или экономика. В v3.2 публичной экономики нет, поэтому soft-shadowban и anomaly detection можно отложить.

### 9.5. Sync strategy

- **Cold start:** pull с сервера актуального кэша + проверка `last_event_id`.
- **Каждые 60 секунд** при активном приложении: push несинхронизированных событий.
- **При Wi-Fi подключении:** форс-sync, если была офлайн-активность.
- **При logout / app kill:** форс-push.
- **После регистрации:** связывание анонимных событий с `auth.users.id`, перенос прогресса.

### 9.8. UX / UI принципы

1. **Dark Mode по умолчанию.** Геймерский вайб, неоновые акценты. Спокойная цветовая палитра (без агрессивного красного/оранжевого).

   **Принцип первого квеста:** максимально низкий порог. Комплимент незнакомцу — это уже уровень 2, его лучше оставить на 2–3 день онбординга.    - Квест 1 (день 1): тёплое «спасибо» персоналу.
   - Квест 2 (день 2): уточняющий вопрос незнакомцу (нейтральный, в магазине).
   - Квест 3 (день 3): короткий комплимент вещи, не человеку.
   - Квест 4+ (с дня 4): полный пул, начиная с level-1.
3. **Прогресс-бар вместо уровней.** Плавный рост XP, без скачкообразных «level up».
4. **Haptic feedback** на каждое важное действие (выполнение квеста, бросок кубика, откладывание).
5. **Calm tone.** Без агрессивного «ПОЕХАЛИ!», «ПОГНАЛИ!», «ТЫ СУПЕР!». Тон — как у спокойного тренера: «Готов попробовать?», «Получилось», «Не вышло — ничего, попробуем ещё раз».

## 10. Аналитика

### 10.1. Принцип

Вместо vanity metrics («цель — 3% конверт») — **kill/continue criteria** с честными порогами. Если метрики ниже порога — останавливаем разработку новых фич и пересобираем core. Если выше — продолжаем и добавляем PRO-фичи.

### 10.2. Core Events

| Событие | Параметры | Зачем |
| --- | --- | --- |
| `app_opened` | `source: push/organic/deeplink` | DAU, retention D1/D7/D30 |
| `onboarding_step_completed` | `step: 1..N` | Дроп-офф в онбординге |
| `quest_started` | `quest_id, level, stat_type, modifier_id` | Какие квесты берут |
| `quest_completed` | `quest_id, xp_gained, time_to_complete, source: daily/boss/shadow` | Воронка успеха |
| `quest_deferred` | `quest_id, defer_reason: not_now, attempt_duration_seconds, had_npc_training, modifier_id` | Какие квесты чаще откладывают и на каком этапе |
| `quest_failed` | `quest_id, fail_reason: timeout/system/legacy` | Технические прерывания и совместимость старых событий |
| `reflection_submitted` | `fear_score, feel_after_score, want_repeat` | Качество квестов |
| `dice_rolled` | `modifier_id, is_pro: bool` | Конверт в PRO через кубик |
| `guide_opened` | `quest_id` | Как часто нужны подсказки |
| `npc_training_completed` | `quest_id` | Сколько юзеров готовятся заранее |
| `boss_raid_participated` | `boss_quest_id` | Вовлечённость в global challenge |
| `pro_trial_started` | `trigger` | Что триггерит триал |
| `pro_converted` | `trial_duration_days, plan: monthly/yearly` | Конверт free→paid |
| `pro_canceled` | `reason` | Churn из PRO |

### 10.3. Kill/Continue Criteria (через 30 дней после релиза)

**Продолжать, если одновременно выполнены все условия:**

| Критерий | Порог |
| --- | --- |
| D7 retention | ≥25% |
| Среднее выполненных квестов в неделю на активного юзера | ≥3 |
| Доля пользователей, сделавших хотя бы 1 оффлайн-квест | ≥25% |
| Доля пользователей, вернувшихся после откладывания | ≥20% |
| Доля пользователей, открывших paywall или начавших trial | ≥5% |

**Менять или останавливать, если хотя бы одно из:**

- Пользователи делают в основном Shadow-квесты и избегают оффлайна (доля оффлайн <25%).
- Среднее число квестов в неделю <2.
- Откладывание приводит к churn (D7 retention среди отложивших <15%).
- Продукт воспринимается как «кринж-челлендж» (по отзывам / соцсетям / TestFlight feedback).
- Появляются жалобы на квесты от третьих лиц.

### 10.4. Ключевые воронки

**Onboarding:** `app_opened` → `first_quest_started` → `first_quest_completed` → `profile_created`. Цель: ≥60% прохождения от `app_opened` до `profile_created`.

**Конверт в PRO:** `pro_trial_started` → `pro_trial_day_3_active` → `pro_trial_day_7_active` → `pro_converted`. Цель: ≥8% конверта от trial к paid (при достижении kill/continue порогов).

**Качество квеста:** для каждого квеста отслеживаем `defer_rate`, медианное `attempt_duration_seconds` до откладывания и `fear_to_feel_ratio`. Если `defer_rate > 40%` или `fear_score` средний ≥8 при `feel_after_score` ≤4 — квест на пересмотр.

### 10.5. Balance Telemetry

- Распределение XP между статами. Если один стат >60% всех квестов — дисбаланс, надо перебалансировать пул.
- Средний `fear_score` по уровням: level-1 должен быть 3–5, level-2 — 5–7, level-3 — 7–9. Если level-1 в среднем даёт 7 — квесты слишком жёсткие для онбординга.
- Доля `want_repeat = false` по типам квестов. Если >50% — квесты этого типа не нравятся, заменить.
- Длительность сессии: здоровая сессия 3–7 минут. Если <2 мин — пользователь зашёл, посмотрел, ушёл. Если >15 мин — застрял в NPC-тренировках, упрощаем.

## 11. Roadmap Разработки (8–12 недель в зависимости от команды)

Сроки варьируются в зависимости от размера команды. Жёсткие 8 недель реалистичны только при наличии сильного Flutter-разработчика + backend-разработчика + отдельного продакт-райтера для контента. Solo/small team (1–2 человека) закладывайтесь на 10–12 недель — иначе есть риск выпустить сырой sync/paywall/analytics.

| Команда | Срок | Риски |
| --- | --- | --- |
| Flutter senior + backend senior + контент-райтер | 8–10 недель | Минимальные, при условии отсутствия side-quests |
| 1 Flutter + 1 backend (без отдельного контента) | 10 недель | Средние, контент пишет разработчик в конце W7 |
| Solo (1 Flutter full-stack) или small team 2 человека part-time | 12 недель | Высокие, sync/paywall/QA могут смазаться |

Без UGC-инфры, без модерации, без storage для видео — разработка быстрее v2 примерно в 2 раза.

### Шаг 1: Core + SQLite (W1)

> Инициализируй Flutter-проект (Dark Mode). Настрой SQLite (`sqflite`) с таблицами `user_stats`, `quests_pool`, `quest_events`, `completed_quests`. Создай Singleton `GameManager` с методами `addXp()`, `spendEnergy()`, `regenerateEnergy()`, `deferQuest()` (без списания энергии и cooldown, с сохранением подготовки). Реализуй логирование всех действий в `quest_events`.

### Шаг 2: Quest System + UI (W2)

> Сгенерируй JSON-массив на 50 квестов (поля: `level` 1–3, `stat_type`, `energy_cost`, `is_shadow`, `safety_reviewed`, `guide_id`). Напиши логику выдачи 3 случайных квестов в день по локальному времени. Сверстай карточки квестов (текст, стоимость, кнопки «Выполнить» и «Отложить») и HUD персонажа (прогресс-бары статов и энергии).

### Шаг 3: Energy / Defer / Shadow (W3)

> Реализуй класс `ModifierDice` (5 модификаторов: Шепот, Драма, Капибара, Робот, Эхо). Добавь кнопку «Бросить кубик» на карточку квеста (1 бросок/день free). Внедри логику статуса «Затворник»: если энергия = 0, обычные квесты блокируются, выдаются только `is_shadow = true`. Реализуй Safe Defer: без списания энергии и cooldown, с сохранением подготовки и событием `quest_deferred`.

### Шаг 4: NPC Training + Quest Guide (W4)

> Создай экран текстового квест-симулятора на основе `npc_dialogs`. Парсинг `options_json`, отображение сообщений NPC, кнопки выбора. При успешном окончании диалога — бафф +10% XP. Создай экран Quest Guide: 3 шага, примеры фраз, скрипт отказа, low-pressure и advanced варианты. Грузится из `quest_guides`.

### Шаг 5: Boss Raid (Self-report) с Low-Pressure (W5)

> Подключи Supabase. Создай таблицы `daily_boss_quests`, `boss_quest_participants`, `quest_events_cloud`, `user_stats_cloud`, `user_auth_link`. Раз в сутки скачивай глобальный квест и выводи 3-й карточкой на главном экране. **Обязательный UI:** селектор варианта (Normal / Low-pressure) перед кнопкой «Я сделал это». При подтверждении — `boss_quest_participants` insert (UNIQUE constraint), инкремент `participants_count`, начисление ×2 XP (одинаковый для обоих вариантов), логирование в `quest_events` с полем `variant`. Анонимный счётчик участников на карточке.

### Шаг 6: Reflection + Profile (W6)

> Сверстай экран Reflection: 3 вопроса (слайдер страха 1–10, слайдер состояния 1–10, да/нет «хочу повторить»), опциональное текстовое поле «Заметка для себя». Сохрани в `quest_reflections`. Создай экран Профиля: круговые диаграммы распределения XP между статами, текущий уровень, прогресс-бар энергии, бейдж «Затворник». Реализуй отображение истории выполненных квестов с возможностью открыть Reflection повторно.

### Шаг 7: Event Log Sync (MVP-ядро) + Analytics (W7)

> Реализуй sync engine: push несинхронизированных событий из `quest_events` в `quest_events_cloud` каждые 60 сек / при Wi-Fi / при cold start. **Только MVP-ядро:** append-only events, дедупликация по UNIQUE (user_id, id), базовая валидация (`quest_completed` требует предшествующего `quest_started`, `xp_delta` соответствует квесту из пула, `energy_delta` не уводит в минус). Incremental recalculation `user_stats_cloud`. При регистрации — связывание анонимного `quest_events` с `auth.users.id` через `user_auth_link`. Post-MVP фичи (rate limiting, anomaly detection, soft-shadowban, full rebuild) — **не реализуем**, см. раздел 9.4. Подключи Amplitude. Зарегистрируй все Core Events (раздел 10.2). Настрой воронки onboarding и PRO-конверта.

### Шаг 8: Paywall / PRO + QA (W8)

> Реализуй Paywall: StoreKit 2 на iOS, Google Play Billing Library 9.0.0 на Android. Server-side receipt validation через Supabase Edge Functions. Активация PRO-фич: 3 броска кубика в день + 1 reroll на квест, расширенные NPC-тренировки, weekly review, расширенная статистика. Триал 7 дней после первого выполненного квеста. Ручное QA на 5 устройствах (3 iOS + 2 Android). Юнит-тесты на `GameManager` и sync engine. UITests на поток: онбординг → первый квест → reflection → регистрация. Performance audit: холодный старт ≤2 сек, FPS ≥55. Подготовка к TestFlight / Google Play Beta.

### Параллельные треки

- **Контент (W1–W4):** написание 50 Quest Guides продакт-райтером. ~12 часов работы. Если команда без отдельного райтера — контент пишет Flutter-разработчик в W7–W8 (добавит 1 неделю к сроку).
- **Бэкенд (W5–W7):** Supabase схема, Edge Functions, RPC для Boss Raid инкремента. Solo — Flutter-разработчик делает сам, добавит 1–2 недели.
- **QA (W8):** полный цикл. Solo/Small team — закладывайте +1 неделю на буфер.

### Критический путь

Шаги 1 → 2 → 3 → 4 → 5 → 7 → 8. Шаг 6 параллелен с 5. Шаги 1–4 не зависят от бэкенда.

### Что можно выбросить из MVP, если не успеваете

При нехватке времени в следующем приоритете на выбрасывание:

1. **Weekly review** (PRO-фича) — сложный отчёт, можно добавить в v3.2.
2. **Adaptive difficulty** (PRO-фича) — алгоритм подбора квестов. Можно выдать статичный random.
3. **NPC-тренировки для всех квестов** — сделать только для 10 самых сложных квестов, для остальных — только Quest Guide.
4. **Расширенная статистика** — оставить только базовый экран с прогресс-барами.

Core loop (daily-квесты + energy + reflection + Boss Raid + sync) нельзя выбрасывать ни при каких условиях — это ядро продукта.

## 12. Ethical Boundaries / Not Medical Advice

Продукт adjacent к теме социальной тревожности. Это создаёт риск, что пользователи воспримут его как терапевтический инструмент — а мы им не являемся. Раздел фиксирует границы продукта и обязательные disclaimers.

### 12.1. Что мы НЕ делаем

- **Не медицинский продукт.** Confidence Quest не диагностирует, не лечит, не предотвращает социальное тревожное расстройство, социофобию, агорафобию или любые другие психические состояния.
- **Не заменяем терапию.** При клиническом уровне тревоги продукт не может заменить работу с лицензированным психотерапевтом или психиатром.
- **Не собираем медицинские данные.** Reflection-вопросы (страх 1–10, состояние 1–10) — это не диагностический инструмент, это игровой механик для self-awareness. Мы не храним эти данные как медицинские, не передаём их врачам, не используем для диагноза.
- **Не даём медицинских советов.** В Quest Guide и NPC-тренировках нет рекомендаций вроде «если у тебя паническая атака, сделай X». Только скрипты социальных действий.
- **Не работаем в кризисе.** Если пользователь в остром distress-состоянии — продукт не подходит. При панической атаке, суицидальных мыслях, кризисе — нужна профессиональная помощь, не приложение.

### 12.2. Что мы делаем

- Даём игровую обёртку для регулярной практики социальных действий.
- Помогаем заметить собственный прогресс через reflection и статистику.
- Нормализуем право на паузу через Safe Defer без штрафа и глобальной блокировки.
- Предоставляем скрипты и подготовку через NPC-тренировки и Quest Guide.

### 12.3. Обязательные disclaimers в продукте

- **При онбординге** (на 2-м экране, после первого квеста): короткий текст «Это не терапия и не медицинское приложение. Если вам тяжело — обратитесь к специалисту. Подсказки: [ссылка на список кризисных линий по гео]».
- **В настройках** — постоянный раздел «О приложении» с расширенным текстом о границах продукта и ссылками на профессиональные ресурсы (по гео: в РФ — psy-fox.ru / yasno.live, в США — 988 Suicide & Crisis Lifeline, в ЕС — локальные эквиваленты).
- **При D30+ использовании** (если пользователь зашёл 30+ дней подряд): мягкое предложение «Заметили, что занимаетесь регулярно? Если хотите глубже — рекомендуем поговорить с психологом. [Ссылка]».

### 12.4. Этические принципы контента

- Квесты не должны формулироваться как «преодолей свою тревогу» или «победи страх». Только «попробуй» / «посмотри, что получится» / «готов?».
- Quest Guide не должен использовать терапевтическую терминологию (CBT, exposure therapy, cognitive restructuring). Только игровые и бытовые формулировки.
- NPC-тренировки не должны имитировать терапевтический диалог. Это симуляция бытового разговора, не сессия с психологом.
- Если квест в Quest Guide не может быть выполнен пользователем по любой причине — пользователь всегда может нажать «Отложить» без списания энергии и блокировки других квестов.

### 12.5. Право на отказ

- Пользователь может пропустить любой квест без объяснения причин.
- Пользователь может отключить push-уведомления в любой момент.
- Пользователь может удалить аккаунт и все данные в любой момент через настройки приложения.
- Пользователь может написать в поддержку с жалобой на конкретный квест — квест будет рассмотрен и при необходимости удалён из пула.

### 12.6. Data Deletion API

MVP должен поддерживать удаление аккаунта и данных без обращения в поддержку:

- **В приложении:** Settings → «Удалить аккаунт и данные» → подтверждение действия.
- **На сервере:** Supabase Edge Function `delete_user_data` удаляет или анонимизирует `quest_events_cloud`, `user_stats_cloud`, `boss_quest_participants`, `user_auth_link` и профиль пользователя.
- **Локально:** после успешного server-side удаления клиент очищает SQLite (`user_stats`, `completed_quests`, `quest_events`, `quest_reflections`) и сбрасывает локальный anonymous id.
- **Аналитика:** после удаления не остаётся пользовательских идентификаторов; допускаются только агрегированные, неидентифицирующие метрики.

## 13. Out of Scope (Не в MVP)

Явное определение того, что **не входит в MVP**. Эти фичи могут прийти в v2/v3, но не должны блокировать релиз MVP.

| Фича | Причина отсрочки | Условие возвращения |
| --- | --- | --- |
| UGC-видео | Юридические риски, стоимость модерации, риск App Store rejection | Доказанный D30 retention ≥25% и явный запрос от пользователей |
| Публичная Arena | То же, что UGC-видео | Только после успешного запуска UGC-видео |
| Лайки / комментарии | Социальная нагрузка, несовместимая с приватным продуктом | Если пользователи сами попросят social layer |
| Hall of Fame | Ранжирование поощряет toxic behavior | Никогда в текущем позиционировании |
| Видео-модерация | Не нужна без UGC-видео | Только если вернётся UGC |
| Moderation dashboard | Не нужен без модерации | Только если вернётся UGC |
| Leaderboards | Любые ранжирования поощряют toxic behavior | Никогда в текущем позиционировании |
| Clans / группы | Требует критической массы, переусложняет онбординг | После 10K DAU |
| PvP (charisma duels) | Реалтайм-инфра, выходит за рамку продукта | Никогда |
| AR / VFX-маски | GPU-рендеринг, дорого | v3.0+ |
| LLM-генерация квестов | Стоимость API, риск safety-нарушений | Только после валидации статичного пула |
| Игровой класс «Тролль» | Риск App Store rejection (Guideline 1.1) | Никогда |
| Публичные профили | Privacy | Если пользователи сами попросят |
| Push-уведомления с упоминанием других пользователей | Privacy | Никогда |

### 13.1. Принцип отсрочки

Каждая фича из таблицы — хорошая идея в вакууме. Но каждая добавляет 2–4 недели к разработке и риск rejection. Принцип: **сначала валидируем core loop (3 стата × 50 квестов × Boss Raid × Reflection) с реальными юзерами, потом добавляем фичи на основе данных**. Если D7 retention <25% — никакая фича не спасёт. Если ≥25% — любая PRO-фича усилит рост.
